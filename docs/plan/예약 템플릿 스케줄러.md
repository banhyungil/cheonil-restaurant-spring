# 예약 템플릿 스케줄러 구현 방안

`m_order_rsv_tmpl` 의 활성 템플릿을 주기적으로 스캔해서 `t_order_rsv` 인스턴스를 자동 생성.

> 핵심 정책: **예약 시간(`rsv_time`) 의 1시간 전** 에 예약 주문을 생성한다.

---

## 1. 트리거 정의

```
template.rsv_time 가 12:30 → 11:30 에 t_order_rsv 생성
template.rsv_time 가 19:00 → 18:00 에 t_order_rsv 생성
```

조건 (모두 만족):

- `active = true`
- 오늘 요일이 `day_types` 에 포함
- `start_dt ≤ today` AND (`end_dt IS NULL` OR `end_dt ≥ today`)
- `rsv_time = (지금 + 1시간)` (분 단위 일치 여부는 §2 참고)

---

## 2. 스케줄 주기 옵션

| 옵션           | 주기            | 정확도               | 부하                             | 비고                                     |
| -------------- | --------------- | -------------------- | -------------------------------- | ---------------------------------------- |
| **A. 매 분**   | `0 * * * * *`   | 분 단위 (≤ 1분 오차) | active 템플릿 SELECT × 60회/시간 | 가장 정확. POS 환경에 적합               |
| **B. 매 5분**  | `0 */5 * * * *` | 5분 단위             | 12회/시간                        | 분 정밀 불필요 시                        |
| **C. 매 시간** | `0 0 * * * *`   | 1시간 windowing      | 24회/일                          | rsv_time 이 시 단위로 정렬되어 있을 때만 |

→ **권장: A (매 분)**.

- active 템플릿 수 적고 (매장 수 × 평균 1~3 템플릿) SELECT 부담 무시 가능
- 1시간 전 정확 도달 보장 → "30분 전에 생성됨" 같은 어긋남 없음

---

## 3. 멱등성 보장

같은 템플릿 + 같은 rsv_at 으로 두 번 생성되면 안 됨.

### A. DB UNIQUE 제약 (권장)

```sql
ALTER TABLE t_order_rsv
  ADD CONSTRAINT uk_order_rsv_tmpl_at
  UNIQUE (rsv_tmpl_seq, rsv_at);
```

- DB 가 마지막 안전망
- INSERT 중복 시 PG 가 `unique_violation` (SQLState 23505) throw → 코드에서 catch

### B. 코드 사전 체크

```java
if (orderRsvRepo.existsByRsvTmplSeqAndRsvAt(tmplSeq, rsvAt)) continue;
```

- 일반 경로에서 중복 INSERT 회피
- A + B 병행 → 일반 흐름은 코드, 동시성 사고는 DB 제약이 잡음

→ **A + B 둘 다** 적용 권장. 단 DB 제약 추가 안 할 거면 B 만으로도 단일 노드 환경에선 충분.

---

## 4. 컴포넌트 구조

```
orderRsvTmpl/scheduler/
├── OrderRsvSchedulerJob.java       @Scheduled cron — Service 호출만
└── OrderRsvSchedulerService.java   핵심 비즈니스 로직 (멱등성, 트랜잭션)
```

### 분리 이유

- **Job**: cron 트리거 정의 + 호출 (단순 wrapper)
- **Service**: 트랜잭션, 비즈니스 로직, 단위 테스트 가능 (cron 의존 없이 직접 호출)
- 수동 trigger endpoint 도 같은 Service 메서드 호출

## 5. 핵심 로직

### Job

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class OrderRsvSchedulerJob {

  private static final long LEAD_MINUTES = 60;   // 1시간 전 트리거

  private final OrderRsvSchedulerService service;

  /** 매 분 정각 실행 — (현재 + 1시간) 시각이 rsv_time 인 템플릿 처리. */
  @Scheduled(cron = "0 * * * * *")
  public void tick() {
    OffsetDateTime now = OffsetDateTime.now().withSecond(0).withNano(0);
    OffsetDateTime targetAt = now.plusMinutes(LEAD_MINUTES);
    int created = service.generateForTargetAt(targetAt);
    if (created > 0) log.info("[rsv-scheduler] generated {} reservations for {}", created, targetAt);
  }
}
```

### Service

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderRsvSchedulerService {

  private final OrderRsvTmplRepo tmplRepo;
  private final OrderRsvTmplMenuRepo tmplMenuRepo;
  private final OrderRsvRepo orderRsvRepo;
  private final OrderRsvMenuRepo orderRsvMenuRepo;

  /**
   * targetAt 시각의 예약 인스턴스 생성. 트랜잭션 단위는 "한 분 batch".
   * 한 템플릿 실패해도 나머지 진행 — try/catch 로 감싸기.
   */
  @Transactional
  public int generateForTargetAt(OffsetDateTime targetAt) {
    LocalDate date = targetAt.toLocalDate();
    LocalTime time = targetAt.toLocalTime();
    String today = toDayCode(date.getDayOfWeek());

    List<OrderRsvTmpl> tmpls = tmplRepo.findActiveMatching(date, today, time);
    if (tmpls.isEmpty()) return 0;

    int created = 0;
    for (OrderRsvTmpl tmpl : tmpls) {
      try {
        if (createFromTmpl(tmpl, targetAt)) created++;
      } catch (Exception e) {
        log.error("[rsv-scheduler] failed for tmpl={}", tmpl.getSeq(), e);
      }
    }
    return created;
  }

  /** @return true = 새로 생성됨, false = 이미 존재 (skip) */
  private boolean createFromTmpl(OrderRsvTmpl tmpl, OffsetDateTime targetAt) {
    if (orderRsvRepo.existsByRsvTmplSeqAndRsvAt(tmpl.getSeq(), targetAt)) return false;

    OrderRsv rsv = new OrderRsv();
    rsv.setStoreSeq(tmpl.getStoreSeq());
    rsv.setRsvTmplSeq(tmpl.getSeq());
    rsv.setAmount(tmpl.getAmount());
    rsv.setRsvAt(targetAt);
    rsv.setStatus(RsvStatus.RESERVED);
    OffsetDateTime now = OffsetDateTime.now();
    rsv.setRegAt(now);
    rsv.setModAt(now);
    OrderRsv saved = orderRsvRepo.save(rsv);

    List<OrderRsvTmplMenu> tmplMenus = tmplMenuRepo.findByIdRsvTmplSeq(tmpl.getSeq());
    List<OrderRsvMenu> rsvMenus =
        tmplMenus.stream()
            .map(tm -> {
              OrderRsvMenuId id = new OrderRsvMenuId();
              id.setMenuSeq(tm.getId().getMenuSeq());
              id.setRsvSeq(saved.getSeq());

              OrderRsvMenu m = new OrderRsvMenu();
              m.setId(id);
              m.setPrice(tm.getPrice());
              m.setCnt(tm.getCnt());
              return m;
            })
            .toList();
    orderRsvMenuRepo.saveAll(rsvMenus);

    return true;
  }

  private String toDayCode(java.time.DayOfWeek dow) {
    return dow.name().substring(0, 3);   // MONDAY → MON
  }
}
```

### Repo 추가 메서드

```java

// OrderRsvTmplRepo — active + 오늘 요일 포함 + 날짜 범위 + 시각 일치
@Query("""
    select t from OrderRsvTmpl t
    where t.active = true
      and t.startDt <= :date

      and array_position(t.dayTypes, :day) is not null
```

## 6. 수동 trigger endpoint (운영/디버깅)

```java
// OrderRsvSchedulerController (또는 OrderRsvController 에 추가)
@PostMapping("/order-rsvs/generate")
public Map<String, Integer> manualGenerate(
    @RequestParam @DateTimeFormat(iso = DateTimeFormat.ISO.DATE_TIME) OffsetDateTime targetAt) {
  int created = service.generateForTargetAt(targetAt);
  return Map.of("created", created);
}
```

용도:

- 새벽 다운타임으로 누락된 예약 복구
- 신규 템플릿 즉시 테스트
- 운영자 도구

멱등성 덕에 **여러 번 호출해도 안전**.

## 7. 동시성 / 분산

### 단일 노드

`@Scheduled` 같은 메서드는 자동 직렬 실행 (이전 fire 끝날 때까지 다음 안 시작) → 동시 실행 0.

### 다중 노드

각 노드가 독립적으로 cron 실행 → 같은 템플릿을 두 노드가 동시에 처리 시도 가능.

- DB UNIQUE 제약이 막음 (한 노드만 성공, 다른 건 unique_violation → catch)
- 더 깔끔한 해결: **ShedLock** 도입 (한 노드만 실행)

→ 현 단계 단일 노드 → §3 의 UNIQUE + 코드 체크면 충분.

---

## 8. 트랜잭션 경계

옵션:

- **A. 한 분 batch 전체 1 트랜잭션** (현재 코드) — 한 템플릿 INSERT 실패 시 전체 롤백 위험
- **B. 템플릿별 1 트랜잭션** — 한 템플릿 실패해도 나머지 진행

§5 의 try/catch 로 A 안에서도 한 템플릿 실패를 흡수하지만, **failed transaction 안의 다른 INSERT 가 영향받을 수 있음** (PG 의 transaction abort 룰 — 한 statement 실패 시 전체 트랜잭션 abort).

→ 안전하게는 **B** 권장. 적용 방법: `createFromTmpl` 메서드를 별도 빈으로 분리 + `@Transactional(propagation = REQUIRES_NEW)`. 코드 약간 복잡해짐.

→ 단순화 위해 A 로 시작, 운영 중 실패 빈도 보고 B 로 전환 검토.

## 10. 단점 / 알려진 한계

| 항목                                          | 영향                                          | 대응                                                                                                              |
| --------------------------------------------- | --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **rsv_time 1시간 전 트리거 누락** (서버 다운) | 그날 예약 누락                                | 수동 trigger endpoint 로 복구                                                                                     |
| **시간대 (timezone)**                         | 매장이 다른 timezone 이면 잘못 계산           | 현재는 서버 timezone 기준. 다중 timezone 도입 시 store.timezone 컬럼 추가                                         |
| **start_dt 과거 기준 무한 backfill**          | 1년 전 시작 템플릿 → 예전 날짜 backfill 시도? | 스케줄러는 "현재+1시간" 만 처리하므로 과거 backfill 안 일어남. 수동 trigger 만 위험 — 미래 시각만 받게 validation |
| **end_dt 종료 후에도 active=true**            | 과거 종료 템플릿이 active 면 무한 누적?       | 매칭 쿼리에 `end_dt >= today` 포함 → 자동 skip                                                                    |
| **분 단위 정확도**                            | 서버 시계 어긋나면 오차                       | NTP 동기화 가정. 검증 시 서버 timezone/시각 확인                                                                  |

## 11. 단계별 구현 순서

1. **Repo 메서드 추가** — `findActiveMatching`, `existsByRsvTmplSeqAndRsvAt`, `findByIdRsvTmplSeq`
2. **`OrderRsvSchedulerService.generateForTargetAt`** 구현
3. **`OrderRsvSchedulerJob`** + `@Scheduled` 등록
4. **수동 trigger endpoint** 추가
5. **(선택) DB UNIQUE 제약** 추가
6. **검증** — 수동 trigger 로 다양한 시나리오 테스트
   - 일치하는 템플릿 1개 → 1개 생성 확인
   - 같은 시각 두 번 호출 → 중복 생성 안 됨 확인
   - active=false → skip 확인
   - 오늘 요일 미포함 day_types → skip 확인
   - 날짜 범위 밖 → skip 확인
7. **(이후) ShedLock** — 다중 노드 운영 시점
